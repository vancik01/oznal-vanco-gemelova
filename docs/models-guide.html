Models & Concepts Guide

Plain-English explanations of every algorithm, metric, and concept used in the OZNAL capstone. Each section tells you what it is, how it works, and exactly how we use it in the LoL early-game prediction project.

Logistic Regression Classifier Done

analysis.Rmd · Sec. S1

A linear model that outputs a probability between 0 and 1. Our interpretable baseline.

What it is

Logistic Regression (LR) models the log-odds of the positive class as a linear combination of features. If golddiffat15 goes up by 1 standard deviation, the log-odds of winning change by a fixed amount — the coefficient.

P(win) = sigmoid(β₀ + β₁·golddiffat15 + β₂·firsttower + ... + βₙ·xₙ)

How it works

  1. Start with random coefficients β
  2. Plug features in, compute predicted win probability via sigmoid
  3. Measure loss (log-loss) against actual result
  4. Adjust coefficients using gradient descent to reduce loss
  5. Repeat until convergence
We use it as the interpretable baseline. Since features are standardized, bigger absolute coefficients = stronger influence on the win probability. A positive golddiffat15 coefficient means gold leads increase win probability (obvious, but the magnitude tells us how much).

Hyperparameters we tune

ParamWhat it controlsOur value
methodWhich optimization backend caret uses"glm" (binomial)
CV foldsHow we estimate out-of-sample performance5

Strengths

  • Fully interpretable — coefficients = feature impact
  • Fast to train, easy to productionize
  • Calibrated probabilities out of the box

Weaknesses

  • Assumes linearity in log-odds — misses interactions
  • Hurts under multicollinearity (rank-deficient warning)
  • Can't model "first tower combined with gold lead" patterns

Random Forest Classifier Done

analysis.Rmd · Sec. S1

An ensemble of many decision trees, each trained on a random subset of data and features. Their votes are averaged.

How it works

  1. Bagging: bootstrap sample the training data N times (with replacement) → N different datasets
  2. Grow a deep decision tree on each — but at each split, only consider a random subset of mtry features
  3. To predict: every tree votes → majority wins (classification)
Each tree votes independently; majority class wins. Averaging de-correlates errors.
We train 500 trees with 5-fold CV across 5 values of mtry (25 fits total). RF is our first non-linear model — it can learn that "first tower + gold lead > 1000" is much more predictive than either alone.

Hyperparameters we tune

ParamWhat it controlsOur value
ntreeNumber of trees in the forest500
mtryFeatures considered per split (tuned)5 values in CV grid

Strengths

  • Captures non-linear patterns and interactions
  • Built-in feature importance (Gini decrease)
  • Robust to outliers and scale

Weaknesses

  • Slower than LR, memory-hungry
  • "Importance" biased toward high-cardinality features
  • Individual predictions harder to explain

XGBoost Classifier Done

analysis.Rmd · Sec. S1

Gradient-boosted trees. Builds trees sequentially, each correcting the errors of the previous. Usually the strongest classifier.

RF vs XGBoost — the key difference

  • Random Forest: all trees built in parallel, independent, vote at the end
  • XGBoost: tree 1 makes errors → tree 2 is trained specifically to fix those errors → tree 3 fixes what tree 2 got wrong → and so on

It's like a relay where each runner is told exactly where the previous one fumbled.

How it works

  1. Fit a simple model (constant prediction = class mean)
  2. Compute residuals (how wrong we are on each row)
  3. Train a shallow tree to predict those residuals
  4. Add the tree's predictions to the running total (scaled by learning rate eta)
  5. Repeat nrounds times
We tune over a grid of max_depth, eta, and nrounds with 5-fold CV. XGBoost typically wins on accuracy but loses on interpretability — feature importance is given in terms of gain (how much a feature improves the loss when used in a split).

Hyperparameters we tune

ParamWhat it controlsTypical effect
max_depthMax tree depthHigher → more complex patterns, risk overfit
etaLearning rate (shrinkage)Lower → slower but more stable learning
nroundsNumber of boosting iterationsMore trees → better fit, risk overfit
subsampleRow sampling per treeAdds regularization

Strengths

  • Usually the highest accuracy on tabular data
  • Built-in regularization (L1/L2) + tree pruning
  • Handles missing values natively

Weaknesses

  • Many hyperparameters → longer tuning
  • Harder to explain than LR or CART
  • Can overfit without careful CV

K-Nearest Neighbors (KNN) Partitioning TODO Adriana

analysis.Rmd · Sec. S1

No training. To classify a new point, find its K closest neighbors in the training set and let them vote.

How it works

  1. Store the entire training set (no "training" phase)
  2. For a new game to classify: compute distance to every training row
  3. Pick the K closest ones
  4. Majority class among those K = prediction
With K=5, we count the 5 nearest points and take a majority vote.
KNN is distance-based — so features MUST be standardized first (otherwise goldat15 in the 20,000s will dominate void_grubs in the 0–6 range). That's why we prepare X_train_scaled specifically for KNN and LR.
We tune k with CV (odd values to avoid ties). Low k = flexible but noisy; high k = smooth but misses local patterns. KNN gets added to the ROC overlay alongside LR/RF/XGB for direct comparison.

Strengths

  • No training — simple mental model
  • Captures arbitrary non-linear boundaries
  • Good baseline for partitioning

Weaknesses

  • Slow at prediction time (scans all training data)
  • Suffers in high dimensions (curse of dimensionality)
  • No intrinsic explainability

Decision Tree — CART Partitioning TODO Adriana

analysis.Rmd · Sec. S1

A single tree of if/else rules. Each split carves the feature space into rectangles.

How it works

  1. At each node, try every possible split on every feature
  2. Pick the split that maximally reduces impurity (Gini or entropy)
  3. Recurse on each child
  4. Stop when nodes are pure or size limit hit
if golddiffat15 > 1500 → Win (92%)
else if firsttower = 1 → Win (71%)
else if xpdiffat15 > 500 → Win (58%)
else → Loss (68%)
CART is our most visually interpretable model. We plot the tree with rpart.plot — every rule is directly readable. Tree-based, so scale-invariant — we use the unscaled X_train.

Hyperparameters we tune

ParamWhat it controls
cpComplexity penalty — smaller = deeper tree
maxdepthMax levels allowed
minsplitMinimum rows needed to attempt a split

Strengths

  • Visual — every decision rule is legible
  • Scale-invariant, handles mixed types
  • Fast prediction

Weaknesses

  • High variance — small data changes = different tree
  • Weaker accuracy than ensembles (RF, XGB)
  • Can overfit deep-branch leaves

RFE — Recursive Feature Elimination Feature Selection TODO Adriana

analysis.Rmd · Sec. S3

Start with all features. Drop the least important one. Re-fit. Repeat until you reach the target count.

How it works

  1. Train model on all 46 features → get importance ranking
  2. Drop the bottom feature
  3. Retrain on the remaining 45
  4. Repeat until you hit the target feature count (e.g. 10)
  5. Report features surviving + performance curve
We use Logistic Regression as the RFE "estimator". At each step, LR coefficients rank features → lowest |β| gets dropped. Output: a ranked list + an accuracy-vs-features curve showing where the sweet spot is.

Strengths

  • Simple, interpretable process
  • Respects the chosen model's logic
  • Shows performance at every step

Weaknesses

  • Expensive — N refits for N features
  • Greedy — can miss feature combinations
  • Dependent on estimator choice

Forward Stepwise Selection Feature Selection TODO Adriana

analysis.Rmd · Sec. S3

Opposite of RFE. Start empty. Add the best feature. Repeat until adding more doesn't help.

How it works

  1. Start with zero features; baseline model = class prior
  2. For every candidate feature, fit a model with it + current set; pick the one that reduces loss most
  3. Add that feature; repeat
  4. Stop when no remaining feature meaningfully improves the model (AIC / BIC / CV drop)
Forward selection is greedy — it picks feature k conditional on features 1..k−1. It may miss a pair of features that only help together. LASSO handles such cases better.

LASSO (L1 regularization) Feature Selection Done

analysis.Rmd · Sec. S3 (line 1031)

Logistic Regression with a penalty that can shrink coefficients all the way to exactly zero. Acts as classifier + feature selector in one step.

The math, in English

Normal LR minimizes: log-loss. LASSO minimizes: log-loss + λ · Σ|βᵢ|. The extra term penalizes large coefficients — and crucially, the L1 norm (sum of absolute values) tends to produce exactly-zero solutions for weak features.

What λ does

  • λ = 0: no penalty — same as ordinary LR, all features kept
  • λ small: a few weak features dropped
  • λ large: almost all features zeroed out — only the strongest survive
  • λ → ∞: everything zero, model = class mean
We use cv.glmnet with 5-fold CV to find the best λ. Rather than picking the minimum CV error (lambda.min), we use lambda.1se — the largest λ within 1 standard error of the minimum. This gives a sparser model that's nearly as accurate but drops more features — ideal for feature selection.

Why it fits our data

Our EDA revealed heavy multicollinearity: goldat15, opp_goldat15, and golddiffat15 carry near-identical signal. LASSO handles this by keeping the strongest of the group and zeroing the rest.

Strengths

  • Does selection + classification in one fit
  • Shrinks coefficients → robust to multicollinearity
  • Sparse output is easy to report

Weaknesses

  • Among correlated features, LASSO picks one arbitrarily — order can depend on data noise
  • Struggles when features > rows (we're fine)

Elastic Net (L1 + L2) Feature Selection TODO Adriana

analysis.Rmd · Sec. S3

LASSO and Ridge combined. Fixes LASSO's "pick one from a correlated group arbitrarily" problem by grouping them together.

The penalty

log-loss + λ · [ α·Σ|βᵢ| + (1−α)·Σβᵢ² ]
  • α = 1 → pure LASSO
  • α = 0 → pure Ridge (shrinks but never exactly zero)
  • α ≈ 0.5 → balanced Elastic Net
When two features are highly correlated (e.g. goldat15 and opp_goldat15), LASSO picks one and zeros the other. Elastic Net tends to keep both with shrunk coefficients, preserving signal while controlling overfitting.
We'll use cv.glmnet with alpha between 0 and 1, tuned via CV. Expected result: a slightly less sparse model than LASSO, potentially with more stable feature selection.

Random Forest Importance Feature Selection TODO Adriana

analysis.Rmd · Sec. S3

Use RF's built-in feature importance as a ranking, then pick the top-N as your selected set.

Two importance types

  • Gini importance (mean decrease in impurity): how much each feature reduces Gini across all splits in all trees. Fast, but biased toward high-cardinality features.
  • Permutation importance: shuffle the feature's values → measure how much accuracy drops. Slower but unbiased and more reliable.
We get RF importance "for free" from the Scenario 1 RF model — just re-use the trained object and rank features. This is the embedded category of feature selection (vs. the wrapper methods RFE and Stepwise).

Confusion Matrix Evaluation

A 2×2 table counting predictions vs actuals. Every other classification metric is derived from it.

Predicted LossPredicted Win
Actual LossTN (true negative)FP (false positive)
Actual WinFN (false negative)TP (true positive)
In our context: TP = we predicted win, team did win. FP = we predicted win, team lost (over-optimistic). FN = we predicted loss, team actually won (missed a comeback-able position). Since the target is 50/50, no class is privileged — we can read all metrics straight off.

Accuracy Evaluation

Fraction of predictions we got right. (TP + TN) / total.

Accuracy is reliable here because our target is perfectly balanced (50/50). A naive "always predict win" baseline gets 50%. Our EDA baseline ("gold lead at 15 → win") hits 72.9% — any model that doesn't beat this isn't learning anything beyond that single rule.
Accuracy is misleading on imbalanced data (e.g. 95% losses → a "always loss" predictor gets 95% accuracy while being useless). Doesn't apply to us, but worth knowing.

Precision, Recall, F1 Evaluation

Three metrics capturing different types of errors.

MetricFormulaAsks
PrecisionTP / (TP + FP)Of the games we predicted as wins, how many actually were?
RecallTP / (TP + FN)Of the actual wins, how many did we catch?
F12·(P·R)/(P+R)Harmonic mean — balances the two
Precision and Recall trade off against each other. A model that predicts "win" only on 90%+ gold leads will have high precision but low recall (misses close games it could have called). A model that predicts "win" on any advantage has high recall but low precision (calls everything a win).

AUC-ROC Evaluation

The area under the ROC curve. Measures how well the model ranks wins above losses, independent of any specific threshold.

What the ROC curve plots

  • X-axis: False Positive Rate (FPR = FP / actual losses)
  • Y-axis: True Positive Rate (TPR = recall = TP / actual wins)
  • Each point = the model's performance at a different classification threshold

AUC interpretation

  • 0.5 — random guessing
  • 0.7–0.8 — fair / good
  • 0.8–0.9 — strong (where our models land)
  • 0.9+ — excellent (suspect data leakage in our domain)
We overlay all 5 models' ROC curves on the same plot in ## Model comparison. A curve closer to the top-left corner = a better ranker. AUC-ROC is our primary metric because it's threshold-independent and handles balanced binary classification cleanly.

Stratified K-Fold Cross-Validation Evaluation

Split training data into K chunks. Train on K-1, validate on the left-out fold. Rotate. Average the K scores.

Why "stratified"?

Each fold preserves the original class balance (50/50). Without stratification, a fold could randomly end up 60/40 or worse, giving biased metrics.

We use 5-fold stratified CV throughout — for LR (1 fit × 5 folds), RF (5 mtry values × 5 folds = 25 fits), XGBoost grid search (many × 5), KNN, CART, LASSO. The trainControl object in caret sets this up once and reuses it across all models so the scores are directly comparable.

Why not a single train/validation split?

Every metric becomes a distribution, not a point estimate. We can report mean ± SD and see if two models' CV scores actually differ beyond noise.

Train/Test Split by Game ID Concept

Each game produces 2 team rows (Blue + Red) that are exact mirrors. We split on gameid, not on row, to prevent leakage.

If we split rows randomly: Blue side of game X could land in train while Red side lands in test. But Blue's golddiffat15 = +500 is literally Red's golddiffat15 = −500 — same information, flipped sign. The model would "cheat" by memorizing games it's already seen.
We split 80% of games to train, 20% to test, stratified by result. The same gameid split is reused for H2 so both hypotheses are evaluated on identical holdout games.

Standardization (z-score scaling) Concept

Transform every feature to mean=0, std=1. Required for distance- and magnitude-based models.

x_scaled = (x − mean(x_train)) / sd(x_train)
Always compute the scaler on training data only, then apply it to test data. If you use test data statistics, you've leaked information about the test set into preprocessing.

Which models need it in our project

ModelScaling needed?Reason
Logistic RegressionYesCoefficients comparable only if features on same scale
KNNYesDistances dominated by large-scale features otherwise
LASSO / Elastic NetYesPenalty applied equally — must equalize scale first
Random ForestNoSplits are threshold-based, scale-invariant
XGBoostNoSame reason as RF
CARTNoSingle tree, scale-invariant

Multicollinearity Concept

When two or more features carry almost the same signal. Breaks linear model interpretation.

In our data: goldat15, opp_goldat15, and golddiffat15 = goldat15 − opp_goldat15 are mathematically redundant. When we initially threw all three into Logistic Regression, R threw the "rank deficient" warning — that's the statistical term for "these columns are linearly dependent, I can't uniquely fit coefficients."

How we handle it

  • Drop the raw absolute values, keep only differentials
  • Use momentum features (diff@15 − diff@10) instead of having both snapshots
  • Rely on LASSO / Elastic Net to auto-zero redundant features
  • Tree-based models (RF, XGB) are tolerant — collinearity doesn't break them, just splits importance arbitrarily between correlated features

Data Leakage Concept

When your training data sneakily contains information about the target that won't be available at prediction time. Produces suspiciously high accuracy that doesn't survive deployment.

Where we defend against it

  1. Feature selection — we reject post-game stats like towers (r=0.89) because they're measured after the game we're trying to predict. Only features available by minute 15.
  2. Train/test split by gameid — so the test set never contains the mirror row of a training game.
  3. Rolling team win rate — we compute each team's historical winrate using only games played before the current one. Teams with no history default to 0.5. Never use the current game's result or any future game.
  4. Standardization fit on train only, applied to test.
A suspiciously good model (AUC > 0.95 on this problem) would almost certainly be leaking. Our models land at AUC ≈ 0.83–0.84 — consistent with the 74–85% range in the published literature, which is a good sanity check.

Feature Engineering Concept

Creating new features that capture signal better than raw columns. Often the single biggest lever for model performance.
