## Critical: Sensitivity/Specificity labels are inverted across the whole doc

`y_test_fct <- factor(y_test, levels = c(0, 1), labels = c("Loss", "Win"))` makes `"Loss"` the first level, so `caret::confusionMatrix` defaults to `positive = "Loss"`. That means:

- `cm$byClass["Sensitivity"]` = recall of **Loss**, not Win
- `cm$byClass["Specificity"]` = recall of **Win**, not Loss

But the print labels say the opposite:

- `analysis.Rmd:841` LR: `"Sensitivity (Win recall)"`, `"Specificity (Loss recall)"` - swapped
- `analysis.Rmd:1054-1059` KNN - same swap
- `analysis.Rmd:1087-1093` CART - same swap
- `analysis.Rmd:1403-1408` RFE - same swap
- `analysis.Rmd:1470-1476` Forward - same swap
- `analysis.Rmd:1548-1556` LASSO - same swap
- `analysis.Rmd:1666-1675` Elastic Net - same swap
- `analysis.Rmd:1740-1744` RF Importance - same swap

Same problem leaks into the comparison table:

- `analysis.Rmd:1115-1126` `calc_metrics` computes `Recall = byClass["Sensitivity"]` and `F1` from it - so the "Recall"/"F1" columns of the model comparison table are **Loss** recall/F1, even though the doc frames the task as "predict Blue wins."

**Fix options:** (a) swap the labels, (b) pass `positive = "Win"` explicitly to `confusionMatrix`, or (c) reorder the factor levels to `c("Win", "Loss")`. Whatever you pick, do it once at the factor creation so every downstream block is consistent.

## Medium: Shiny per-side flag inversion is biased

`analysis.Rmd:1997-2000` for the Red-side row:

```r
firstblood = ifelse(firstblood == 1L, 0L, firstblood)
```

This sends `blue=1 → red=0` correctly, but `blue=0 → red=0`. The mutually-exclusive flags have `blue + red ≤ 1`, not `= 1`, so the "neither secured" case exists - but it's tiny (well under 1% for first blood, which always happens). For `firstblood`/`firsttower` the dominant non-Blue case is *Red secured*, so this collapses Red's actual securer rate to 0. `1L - firstblood` would be ~99% correct vs. the current ~50% correct for the Red row. Better fix: keep `red_firstblood` etc. through the pipeline instead of reconstructing.

This is Shiny-export only, not the main modelling, but the saved `game_lookup` will under-represent Red's first-objective rate.
